## 結論
CI/CDとは、一言でいうと **「システム開発の『作る・テストする・公開する』という流れを、人間がやる代わりにロボット（自動化ツール）に全部やらせること」** 。

## CI/CDの正体
CI/CDは、大きく2つのステップに分かれる。

### CI（Continuous Integration：継続的インテグレーション）
「バラバラに書いたコードを、こまめに（継続的に）合体させてテストする」こと。

やる仕事:
- コードが最新かチェックする
- コンパイルする（JavaならここでJARを作る）
- 自動テストを実行してバグがないか確認する

メリット: 誰かが書いたコードで「動かなくなった！」という事態をいち早く察知できる。

### CD（Continuous Delivery / Deployment：継続的デリバリー/デプロイ）
「テストが終わったコードを、いつでも本番（または検証環境）に出せる状態にする」こと。

やる仕事:
- Dockerイメージを作成する
- 本番サーバーに送り込む（デプロイ）
- 古いバージョンと入れ替える

メリット: 「ボタン一発」あるいは「自動」でリリースできるので、作業ミスが激減する。

## キャッシュがここでどう活きるか？
先ほどの「ライブラリのキャッシュ」の話がここで繋がる。CI/CDツール（GitHub Actions, CircleCI, Jenkinsなど）は、実行のたびに「真っさらなコンピュータ」を用意して作業する。そのままでは、毎回数GBのライブラリをネットから落とすことになり、待ち時間が長くなる。そこで、**「前回のビルドで使ったライブラリ（JARなど）をキャッシュしておき、次回のCIで使い回す」** という設定をする。

- キャッシュなし: ビルドに10分かかる
- キャッシュあり: ビルドが2分で終わる

この「爆速化」こそが、開発者のストレスを減らす鍵。

## 現場で使われる代表的なツール

| カテゴリ | 有名なツール | 特徴 |
| --- | --- | --- |
| プラットフォーム一体型 | GitHub Actions | 今一番人気。GitHubを使っていればすぐ始められる。 |
| クラウド型 | CircleCI | 高速で、設定が柔軟。 |
| オンプレ型 | Jenkins | 昔からある。カスタマイズ性が非常に高い（おじさんエンジニアに人気）。 |

## CI/CDを導入する最大の理由
人間が手動でデプロイすると、必ずいつかミスをする。
- あ、最新のファイルを送り忘れた！
- 設定ファイルを書き換え忘れた！
- テストするのを忘れて、バグがあるまま公開しちゃった！

CI/CDは、これらの「うっかり」を根絶し、エンジニアを単純作業から解放するための仕組み。

## まとめ
- **CI**: 壊れてないか自動でチェックし続ける仕組み。
- **CD**: 自動で（または楽に）本番へ届ける仕組み。
- **キャッシュ**: これらを高速化するための魔法のメモ。
