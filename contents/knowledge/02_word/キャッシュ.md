## 結論
IT現場での「キャッシュ」を一言でいうと、**「重い処理を避けるために、一度手に入れたデータを『近場』にメモしておくこと」** 。

## 活用例
用途によって呼び方や場所が変わる。

### ブラウザキャッシュ (Browser Cache)
- 一番身近なもの
- ウェブサイトを訪れた際、画像やスタイルシート（CSS）を自分のPCに保存する
- 用途: 次に同じサイトを開くとき、ネットから落とさず自分のPCから読み込むので、表示が爆速になる
- 現場の会話: 「デザイン修正したのに変わってないです！」「あ、ブラウザキャッシュ消して再読み込みしてみて」

### サーバーサイド・アプリケーションキャッシュ
- データベース（DB）へのアクセスは時間がかかる。そのため、頻繁に使うデータはメモリ上に置いておく
- 用途: 「都道府県一覧」や「商品ランキング」など、毎回計算したりDBに聞きに行ったりするのが無駄なデータを保持する
- ツール: Redis（レディス）, Memcached などが有名

### CDNキャッシュ (Content Delivery Network)
- 世界中に配置されたサーバーに、画像や動画のコピーを置いておく仕組み
- 用途: 日本のユーザーには日本のサーバーから、アメリカのユーザーにはアメリカのサーバーからデータを返す
- 現場の会話: 「動画配信が重いからCDNのキャッシュ戦略を見直そう」

### ビルド・デプロイ時のキャッシュ
- プログラムをビルド（組み立て）する際、変更がないパーツを再利用
- 用途: Dockerのレイヤーキャッシュや、Maven/Gradleのライブラリキャッシュ
- 現場の会話: 「ライブラリのダウンロードで時間がかかるから、CI/CDのキャッシュを効かせよう」

### CPUキャッシュ
- ハードウェアに近い、もっとも「速い」キャッシュ
- 用途: メモリ（RAM）すら遅く感じるほど超高速なCPUが、演算データを一時保管するために使う（L1, L2キャッシュなど）

## キャッシュを扱う上での「格言」
エンジニアの世界にはこんな有名な言葉がある。

```PhilKarlton
「コンピュータサイエンスにおける難問は2つだけだ。キャッシュの無効化と、命名（名前付け）である。」
```

### なぜ「キャッシュの無効化」が難しいのか？
それは、**「元のデータが変わったのに、古いメモ（キャッシュ）を使い続けてしまう」** という事故が絶えないから。

例: ニュースサイトで記事を更新したのに、キャッシュのせいで古い記事が表示され続けてしまう…など。

## まとめ：キャッシュのメリット・デメリット

| 特徴 | メリット | デメリット |
| :--- | :--- | :--- |
| 速度 | 爆速になる。ユーザー体験向上。 | - |
| コスト | サーバーや回線の負荷が減る。 | - |
| 鮮度 | - | データが古くなるリスクがある。 |
| 複雑さ | - | 仕組みが複雑になり、バグの原因になりやすい。 |

「キャッシュ」という言葉が出てきたら、**「あ、どこかにコピーを置いて横着しようとしてるんだな」** と解釈すれば、だいたい合っている！
