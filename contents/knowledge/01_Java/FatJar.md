## 結論
FatJar（ファットジャー）は、Javaの開発において、実行に必要なすべてのライブラリや依存関係を一つのJARファイルにまとめたものを指す。

通常、Javaのプログラムを動かすには自分のコードだけでなく、外部のライブラリ（ライブラリA、ライブラリB…）が必要になるが、それらを全部ひとつの「太った（Fat）」ファイルに詰め込むことからそう呼ばれている。

別名で Uber-JAR と呼ばれることもある。

## Fatの語源
Fatは「太った」の意味で、技術的な裏付けがある面白い語源（比喩）が含まれている。

### 直感的な比喩：中身がパンパン
Javaの標準的なJARファイルは、自分が書いたプログラム（.classファイル）だけを格納するため、非常に軽量。

一方で、FatJarは以下のものをすべて詰め込む。

- 自分のコード
- 外部ライブラリ（Apache Commons、Log4jなど）
- それらが依存している別のライブラリ
- 場合によっては設定ファイルなど（外だししようぜそこは）

その結果、ファイルサイズが通常の数十倍〜数百倍に膨れ上がり、**「ライブラリをたくさん食べて太ってしまったJAR」** というイメージから「FatJar」と呼ばれるようになった。

巨人師匠：「パンパンやな」

### Uber-JARとの関係
FatJarの正式な呼び方に **Uber-JAR** がある。

- Uber（ウーバー）: ドイツ語の「über（〜の上を、超越した）」に由来する英語のスラング。
- 意味: 「究極の」「超絶な」「すべてを包含する」といったニュアンスです。

つまり、**「依存関係の問題を超越した、これ1つで完結する最強のJAR」** という意味でUber-JARと呼ばれ、その「見た目の重さ」に注目した俗称がFatJarとして定着した。

### 業界のトレンド：痩せる方向へ（Thin JAR）
最近では、FatJarの「重すぎる」「ビルドに時間がかかる」という欠点を解消するために、**Thin JAR（痩せたJAR）** という言葉も使われるようになった。

- FatJar: 全部入り。便利だけど重い。
- Thin JAR: 自分のコードだけ。ライブラリは外部に置いて、実行時にリンクする。

「Fat（太った）」という表現があったからこそ、対義語として「Thin（痩せた）」という言葉が生まれた。

## Fat vs Thin

| 比較項目 | Fat JAR (Uber-JAR) | Thin JAR (Standard JAR) |
| --- | --- | --- |
| 中身 | 自分のコード + 全依存ライブラリ | 自分のコードのみ |
| ポータビリティ | 最強。 ファイル1つでどこでも動く。 | 低い。 実行環境に別途ライブラリが必要。 |
| ビルド時間 | 長い（毎回全ライブラリを固めるため）。 | 短い。 自分のコードだけなので一瞬。 |
| CI/CD効率 | 悪い。1行修正でも巨大なファイルを転送。 | 良い。 修正分（数KB）だけ送れば済む。 |
| Docker相性 | 初心者向き（簡単）。 | 上級者向き。 レイヤー機能を活用できる。 |

## メリット・デメリットの詳細

### Fat JAR が向いているケース
「とにかく楽に動かしたい時」

- メリット: java -jar app.jar と打つだけで動くため、構築ミスが起きにくい。
- デメリット: ライブラリを1つ更新するだけで、数百MBのファイルをサーバーにアップロードし直す必要がある。

### Thin JAR が向いているケース
「プロの開発現場・大規模運用」

- メリット: ライブラリ（依存関係）はサーバー側に共通キャッシュとして持たせておけば、アプリのデプロイは数KB〜数MBの転送で済む。
- デメリット: 実行時に「ライブラリはここにあるよ」というパス指定（-cp オプションなど）を正しく管理する必要がある。

## 結論：どっちを使えばいい？
- 個人開発や小規模なツール: 迷わず Fat JAR。環境構築の手間がゼロになるメリットが勝る。
- Spring Boot などのモダンな開発: 基本は Fat JAR （Executable JAR）が標準だが、Dockerイメージを最適化したい場合は、内部的に Thin化 してレイヤーを分ける手法が取られる。

最近はクラウド（AWS/GCPなど）へのデプロイ頻度が高いため、**「基本はFat JARで作るけど、Dockerの層（Layer）を分けて、ライブラリ部分をキャッシュさせる」** という、いいとこ取りの手法が主流。

現場でPMからFatJarにしようぜって言われた時に、デプロイ時にどのくらいコストがかかるか提案して検証したほうが良さそうだねえ。
